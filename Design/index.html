What We're Building
kllctrs — the first institutional-grade hub for the US trading card industry. Here's exactly what ships in the MVP.

The Problem

The US trading card hobby is a multi-billion dollar industry — but finding a card show, sponsor, or local card shop requires checking 5+ sites, Facebook groups, and Discord servers. The average collector spends 12 minutes to find a single show near them.

Every existing platform has a late-2000s mobile experience. Nobody has invested in a modern, trustworthy, mobile-first product for this market. That's kllctrs.

The One-Line Vision

"Bloomberg of the Trading Card Hobby" Authoritative, data-rich, and deeply native to the hobby's culture — with the mobile-first UX that the hobby has never had.
Reference Aesthetic Heritage Auctions website meets Bloomberg Terminal dark mode meets ESPN dark mode app. Premium data product. Not a startup landing page.

The 6 Features That Ship in MVP

🗓️
Event Calendar & Map
Interactive Google Map showing card shows across the US. State-level zoom, event pins, slide-in drawer with full show details. Filter by state, date, keyword. User show submission with admin approval.
Core Feature
🏪
Card Shop Finder
Map + list view of US card shops. Filter by specialty: Sports Cards / Pokémon / Both. User submissions. Powered by Google Places API. The Yelp for card shops — nothing like it exists.
Core Feature

Filterable directory of major hobby sponsors — PSA, Beckett, Heritage, Topps, Panini, eBay, and more. Each sponsor gets a profile page showing their show history and activation scale. Crucially, sponsors can use the platform to monitor competitors — see which shows rival brands are sponsoring, at what frequency, and in which markets. Combined with profile analytics (who clicked, from which show page), it becomes a marketing intelligence tool for corporate sponsorship teams.
Core Feature

AI Chatbot
Answers questions using the internal database (shows, shops, sponsors) + eBay sold listing prices via official API. Gemini 2.5 Flash. Cites all sources. Has a designated chat interface within the platform.

Content Engine
AI-generated blog posts triggered by new events in the database. LinkedIn auto-posting. Monthly newsletter via MailerLite. Admin approves all content before publishing. Email alerts for users.
Lean MVP

Full command center for the client (Leo). Approve/reject event and shop submissions, edit any listing live, view platform analytics, manage users. Leo operates this independently after launch.

Who It's For
Four distinct user types — each with different goals, devices, and trust signals. Every design decision must serve all four.
🧢
The Casual Collector
Attends 3–5 shows a year. Primarily on mobile. Wants to find shows near them in under 30 seconds. Values clarity and speed. Will leave if the mobile experience is bad — which it is on every competitor site.
Primary User
🏪
The Professional Dealer
Travels to 30+ shows a year. Needs comprehensive data — sponsor history, show size, venue details. Uses desktop more than mobile. Trusts data quality over visual polish, but a premium look signals that the data is serious.
Power User
🏢
The Sponsor / Brand
PSA, Heritage, Topps, Panini, Beckett. They're not just listing on KLLCTBLS to be found — they're using it as a competitive intelligence layer. They can see which shows their competitors are sponsoring, how many events they're activating at, and at what scale — giving corporate teams the data they need to make smarter decisions about which markets to target, how frequent their presence should be, and where they're being outspent. On the flip side, KLLCTBLS gives them first-party data on their own profile: who's clicking, how often, and from which shows. A marketing tool that works from both ends — competitive intelligence outward, audience data inward. That's not a directory listing. That's a strategic platform.
Strategic Partner
📅
The Show Organizer
Runs 1–12 shows per year. Submits their own event listings. Wants their event to look credible and professional on the platform. The submission form needs to feel like a quality tool, not a Google Form.
Data Partner
🏬
The Card Shop Owner
Runs a physical card shop — sports cards, Pokémon, or both. Wants their shop discovered by collectors in their city without paying for ads. Submits their own listing and cares deeply about how their shop profile looks — hours, specialty, photos. Also wants to know when card shows are happening nearby so they can plan inventory and promotions around them.
Local Merchant

The 5 Non-Negotiable Design Principles

# Principle What It Means in Practice
01 Map is the Hero The interactive map is the centrepiece of every discovery screen. On desktop it's always visible. On mobile it's one tap away. Nothing competes with it visually.
02 Mobile is Reality Primary use case: a collector at a show, on their phone, looking up information in real time. Design at 390px first. All key actions reachable by thumb. Minimum 44px touch targets.
03 One Primary Action Per Screen Every screen has exactly one dominant CTA. Secondary actions are visually subordinate. Users should never have to read copy to know what to do next.
04 Trust Through Data Signals "Last updated: 2 days ago" builds more trust than a polished illustration. Source attribution and timestamps are part of the brand voice, not footnotes.
05 Speed is a Feature Skeleton states, not loading spinners. Maps load progressively. No interaction feels slow. Performance is a design requirement with measurable targets (LCP < 2s).
06 Data Signals Build Trust "Last updated: 2 days ago" and "Source: NSCC" are deliberate design elements — not footnotes. Sponsor profile views, competitor show counts, and click-through data must be presented with the same visual weight as the event data. Numbers need to look like they mean something.

Map Pin System — Needs Special Attention

The map is the most-used element on the entire platform The pin design system is one of the highest-impact design decisions in this project. Pins must be readable at all zoom levels, clearly differentiate event types, and feel on-brand — not like default Google Maps pins.
Pin Type Context Must Differentiate From
Default Event Pin All published card shows Must be clearly distinct from shop pins
Featured Event Pin Featured / promoted shows More prominent than default — not just a color change
Hover / Active State Mouse over or tapped Clear affordance that this pin is selected
Shop Pin Card shop locations Different shape or style from event pins
Cluster Pin 3+ pins in same area at zoom level Shows count. Must not look like a regular pin.

Handoff Protocol

1

Design Freeze — Before each sprint

Frames for that sprint are locked in Figma before Rajeev's sprint starts. Rajeev and you walk through the designs live on a call — every component, state, and interaction is confirmed before dev begins.

2

Figma Handoff Mode

Rajeev gets Developer access to the Figma project. All measurements, colors, and CSS properties auto-generated via Figma's inspect panel. Design tokens exported as JSON and shared separately with each handoff.

3

Asset Export — GitHub PR

You export all final assets (logos, icons, images, map pins) into a structured folder. Open a PR on the GitHub repo with assets organized by screen. Rajeev reviews and merges.

4

Implementation QA — Ongoing

You review the staging URL weekly against Figma specs. Raise discrepancies via a shared GitHub issue. Rajeev fixes within 48h. No launch without your pixel-level approval of the staging environment.

Design QA Checklist — Pre-Handoff

  • All screens at 1440px (desktop) and 390px (mobile) — both breakpoints designed for every screen
  • All interactive states present: hover, focus, active, disabled, loading, error, empty
  • Map pins designed at all states: default, hover, active, featured, cluster
  • Custom Google Maps style JSON defined — not default blue/white Google Maps
  • All animations specced with timing, easing, and delay values — not "animate this somehow"
  • Bottom sheet / drawer behaviour specced for both mobile and desktop
  • Font usage is consistent — no heading using body font or vice versa
  • Color contrast verified — especially gold on any non-navy background
  • Touch targets all 44px minimum — documented in Figma annotations
  • OG image template provided — used auto-generated on every event page
  • Icon set exported as SVG — no raster icons anywhere
  • Design tokens exported as JSON — not just a Figma styles panel
